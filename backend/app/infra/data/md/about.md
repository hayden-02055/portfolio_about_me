# About

## 한 줄 소개
문제를 구조적으로 재정의하고, 유지보수 가능한 시스템으로 재설계하는 개발자.

---

## 구조 재설계 경험: CQRS 복잡성 해결

### 1. 문제 상황

- NestJS 기반 CQRS 패턴을 사용하고 있었음
- Command / Query 경계가 명확하지 않았음
- application layer와 infra layer 의존성이 혼재되어 있었음
- 일부 기능 수정 시 관련 의존성 추적에 수 시간 소요
- 특정 로직 수정 과정에서 IDE 상에서 구조 이해가 어려워 일부 디렉터리를 별도 환경으로 분리하여 분석해야 했음

### 2. 기술적 문제의 본질

- 의존성 방향이 단방향으로 정리되어 있지 않았음
- Handler가 여러 계층을 직접 참조하고 있었음
- 도메인 규칙이 application과 infra에 분산되어 있었음
- 에러 발생 시 stack trace가 도메인 의미를 설명하지 못했음

### 3. 결정적 깨달음

AI 기반 바이브코딩을 활용하면 빠르게 코드를 작성할 수 있다.
하지만 구조가 명확하지 않으면 인간이 리뷰할 수 없는 코드가 된다.

결론:
- 생산성은 도구가 만든다.
- 유지보수성은 아키텍처가 만든다.

### 4. 내가 내린 판단

- 디렉터리 구조를 클린 아키텍처 기준으로 재정렬
- 의존성 방향을 interface → application → domain 단방향으로 통일
- infra는 port 구현체로만 존재하도록 제한
- 인증(auth)을 별도 경계로 분리
- AI 서버(FastAPI)와 API 서버 분리
- MSA 형태로 기능 단위 경계 설정

### 5. 조직적 상황

- 개발자들은 구조 명확화에 동의
- 경영진은 단기 개발 속도 저하를 우려
- 안정성 개선 작업은 신규 기능 개발로 보이지 않음

### 6. 트레이드오프

- 단기 생산성 감소
- DevOps 복잡도 증가
- 초기 리팩터링 비용 증가

### 7. 결과

- 기능 수정 시 영향 범위 예측 가능
- 에러 추적 시간 단축
- 코드 리뷰 효율 개선
- 신규 개발 시 구조 충돌 감소

## 아키텍처 철학

- 작동하는 시스템보다 이해 가능한 시스템이 중요하다.
- 구조는 미래의 개발 속도를 결정한다.
- AI 시대일수록 아키텍처의 중요성은 더 커진다.

## 데이터 가공(Data Processing)에 대한 관점

### 1. 모델보다 데이터가 더 중요하다고 생각하는 이유

모델은 확률 분포를 학습한다.
잘못된 데이터는 잘못된 확률을 만든다.

LLM, RAG, 추천 시스템 등 어떤 시스템이든
결과 품질은 입력 데이터의 구조와 품질에 직접적으로 의존한다.

나는 모델 튜닝보다 먼저 다음을 고민한다:

- 데이터가 문제를 정확히 표현하고 있는가?
- 노이즈는 어디서 발생하는가?
- Feature가 도메인 의미를 반영하는가?

---

### 2. 데이터 가공에서 중요하게 생각하는 요소

- 정규화 및 스케일링
- 결측치 처리 전략
- 이상치 탐지
- Feature engineering
- 도메인 기반 파생 변수 생성
- 로그 기반 구조화 데이터 설계

특히 다음 과정에서 재미를 느낀다:

- 원시 로그를 의미 있는 지표로 변환하는 과정
- 정제된 데이터가 모델 성능을 개선하는 순간을 확인하는 것
- 분산 환경(Spark, PySpark)에서 데이터 파이프라인 설계

---

### 3. RAG 관점에서의 데이터 설계

RAG 시스템에서 중요한 것은 단순 벡터화가 아니다.

- 문서를 어떻게 chunking 할 것인가
- 메타데이터를 어떻게 설계할 것인가
- retrieval 시 score threshold를 어떻게 조정할 것인가
- 근거 기반 응답을 어떻게 강제할 것인가

나는 RAG를 단순 LLM 보조 도구가 아니라
데이터 구조 설계 문제로 본다.

---

### 4. 확률 기반 사고와 데이터 처리의 연결

데이터는 모델의 확률 공간을 정의한다.
잘못 가공된 데이터는 모델의 불확실성을 증가시킨다.

따라서 나는 다음을 중요하게 생각한다:

- 분포를 이해하고 처리하는 것
- 학습 데이터와 추론 데이터의 차이를 인식하는 것
- 데이터 누수를 방지하는 설계
- 대규모 데이터 처리 시 shuffle 병목을 고려한 설계

---

### 5. 내가 지향하는 데이터 엔지니어링 방향

- 단순 수집이 아닌 의미 기반 데이터 설계
- 확률 모델과 시스템 아키텍처를 연결하는 데이터 파이프라인
- 모델 성능과 인프라 비용을 동시에 고려하는 설계

## 인공지능에 대한 관점

### 1. AI를 바라보는 기본 태도

나는 인공지능을 단순한 API 호출이 아닌,
확률과 수학 위에서 동작하는 모델로 이해하려고 노력한다.

LLM은 마법이 아니라 확률 분포를 기반으로 다음 토큰을 예측하는 모델이다.
이 원리를 이해할 때, 모델의 한계와 가능성이 동시에 보인다.

---

### 2. 내가 AI에서 즐거움을 느끼는 지점

- 확률 분포가 어떻게 학습되는지 분석하는 과정
- 손실 함수가 수렴하는 과정 관찰
- 모델이 일반화에 실패하는 지점 탐구
- 데이터 품질이 모델 출력에 미치는 영향 분석

특히 다음 영역에 흥미를 느낀다:

- 확률론 기반 모델링
- 최적화 문제
- 분산 학습 구조
- 대규모 데이터 처리(Spark, PySpark)
- LLM 아키텍처와 추론 구조

---

### 3. 엔지니어링과 연구 사이의 균형

현재 나는 두 영역을 동시에 탐색하고 있다.

- 프로젝트에서는 시스템을 안정적으로 만드는 엔지니어 역할
- 학습과 실험에서는 모델 내부를 이해하는 연구자적 접근

나는 모델을 단순히 사용하는 개발자가 아니라,
모델의 동작 원리를 이해하고 시스템과 결합하는 개발자가 되고 싶다.

---

### 4. AI 시대에 대한 생각

AI는 생산성을 극단적으로 높인다.
하지만 구조가 없는 생산성은 장기적으로 기술 부채가 된다.

나는 다음을 중요하게 생각한다:

- 모델의 불확실성을 이해하는 것
- 확률적 시스템을 deterministic 시스템과 어떻게 연결할지 고민하는 것
- LLM을 사용할 때 RAG, 구조화된 출력, 평가 체계를 함께 설계하는 것

---

### 5. 내가 지향하는 방향

- 확률과 수학을 기반으로 모델을 이해하는 엔지니어
- 구조를 설계할 줄 아는 AI 개발자
- 단순히 AI를 붙이는 사람이 아니라,
  AI를 시스템 안에서 안전하게 작동시키는 사람
